# Textbook Generator v3 — Foundation Proposal

## Purpose

This proposal covers the foundational pieces required before any other
v3 work begins. These are not the first tasks of many — they are the
prerequisite layer. Nothing downstream is buildable without them.

**Scope:** Data contracts · Persona example files · Lens library ·
Signal Extractor · Clarification loop · Lesson Architect node ·
WorkOrder compiler · Surgical repair definition

**Out of scope:** Executor restructure · Coherence reviewer · Frontend
collapse · Staging environment (separate proposal)

**Codebase:** Textbook Generator backend only. Lectio v3 is complete and
consumed as a dependency. No Lectio changes in this proposal.

---

## V3 Recreation Principle

v3 is built alongside v2, not on top of it. The v2 pipeline stays running
and untouched during v3 development. Switchover happens only once the
Lesson Architect is producing valid blueprints on real teacher briefs.

The rule for each existing node:

```
If a v2 node can be cleanly repurposed by reading a WorkOrder
instead of its current input → restructure it (later, separate proposal).

If a v2 node's logic is entangled with the old pipeline shape
or would require significant surgery → recreate it in v3 directories
and leave the v2 node untouched until v3 is stable.
```

Concretely:

```
Recreate (new — lives in src/v3_blueprint/ or src/architect/):
  Signal Extractor         — new concept entirely
  Lesson Architect         — new concept entirely
  Blueprint compiler       — new concept entirely
  Coherence reviewer       — new logic, blueprint-aware
  Surgical repair handler  — new concept entirely

Restructure later (existing node, reads WorkOrder instead):
  content_generator → section writer
  diagram_generator → visual executor

Retire when v3 is stable:
  curriculum_planner
  composition_planner
  qc_agent
```

All new v3 code lives under:

```
backend/src/v3_blueprint/
backend/src/architect/
backend/resources/lenses/
```

No v2 files are modified in this proposal.

---

## Context

Lectio v3 is complete. Every component now exports `phase`, `role`,
`cognitiveJob`, `capabilities`, `teacherLabel`, and `teacherDescription`
via `manifest.json`. The Textbook Generator consumes this JSON without
importing Lectio source code.

The current Textbook Generator pipeline runs:

```
curriculum_planner → content_generator → composition_planner
→ diagram_generator ‖ interaction_generator → section_assembler → qc_agent
```

Each node sees a slice. No node holds the whole lesson. This produces
locally coherent but globally inconsistent output — the problem v3 solves.

v3 inserts a Lesson Architect above the pipeline that holds the entire
lesson in mind simultaneously, produces a Production Blueprint as the
single source of truth, and compiles that blueprint into WorkOrders that
each executor reads. No executor improvises. No executor contradicts another.

This proposal delivers the first three pieces that make that possible.

## Piece 1 — Data Contracts and Persona Example Files

### What it is

Typed Pydantic models for every object in the v3 pipeline, plus four
hand-written example blueprint JSON files — one per persona. These are
written by a human, not generated by an LLM. They prove the schema
captures real pedagogical decisions before any generation is attempted.

### Why it comes first

Do not start with prompts. Start with typed objects. The schema is the
contract the entire system runs on. Get it right here and everything
downstream has a stable target. Get it wrong and everything needs reworking.

The example files are equally important. They encode the correct blueprint
output for each persona scenario. When the Lesson Architect is built in
Piece 3, these files are the ground truth it is tested against.

## Piece 2 — Lens Library

### What it is

A library of YAML files encoding pedagogical wisdom. The Lesson Architect
reads the full library before planning any lesson. Lenses replace the rigid
five pack types (introduce / practice / reteach / assess / differentiate).

A lesson can draw from multiple lenses simultaneously. The agent selects
which lenses apply from teacher signals and applies their principles
together when planning.

### Why it comes first

Without lenses the Lesson Architect has no pedagogical reasoning foundation.
It is a capable model with a blank brief. With lenses it has accumulated
teaching wisdom to reason against. The lenses are also the fastest piece to
produce — no code, no infrastructure, pure content work.

### Source material

The existing pack spec YAML files contain strong `intent` and `plan_guidance`
blocks. Mine these directly. The wisdom is already written — it needs
reframing from rigid templates into reasoning principles.

Files to mine:
```

backend/resources/packs/assess.yaml

```

### YAML schema per lens

```yaml
id: first_exposure
label: First Exposure
category: lesson          # lesson | support
applies_when: >
  The learner has no working model of this concept.
  This is genuine first teaching, not review.

reasoning_principles:
  - Create need before formal explanation.
  - Use one stable anchor example throughout.
  - Model before independent practice.
  - Move warm to medium before attempting cold.
  - Never ask a learner to practise what they have not first understood.

blueprint_effects:
  visual_density: medium_or_high
  practice_progression: warm_to_medium
  vocabulary_policy: define_first
  anchor_reuse: entire_resource
  cold_questions: avoid_for_below_average

principles:
  anchor: >
    Choose one concrete example and reuse it everywhere — explanation,
    worked example, and the first practice questions. Familiarity before
    variation. Consistency reduces cognitive load on first exposure.

  sequence: >
    Create need before building knowledge. Never ask a learner to practise
    what they have not first understood. The sequence is not negotiable on
    first exposure: orient → explain → model → practise.

  visual: >
    Spatial and procedural concepts need visual support. Do not skip it.
    Reuse the anchor shape across visuals so the learner builds a stable
    mental image. Step-sequence visuals show one step at a time.

  difficulty: >
    Warm questions use the anchor example with minimal variation.
    Medium questions introduce one dimension of variation only.
    Cold questions ask for transfer. Do not reach cold until the learner
    has succeeded at warm and medium. For below-average groups on first
    exposure, cold may not appear at all.

  adaptation: >
    Below average: reuse anchor throughout, whole numbers only, more
    worked steps, diagrams on all questions including medium.
    Average: anchor in explanation and first two questions, new shapes at cold.
    Above average: anchor in explanation only, varied contexts throughout,
    transfer question required.

avoid:
  - assuming prior fluency with this concept
  - starting with cold or transfer questions
  - different examples across explanation and practice

conflicts_with:
  - consolidation
  - transfer
```

### All eight lenses — content summary

**Lesson lenses:**

`first_exposure` — anchor example everywhere, create need before practice,
warm → medium → cold progression, no cold for below-average on first
exposure, visual support required for spatial/procedural concepts.

`consolidation` — different examples from original teaching so transfer is
genuine, start at medium difficulty not warm, cold and transfer questions
are the target, minimal worked example (one at most), answer key with full
working.

`repair` — target the fault line only, not the whole concept again,
different approach and different example from original teaching, tight
practice set targeting only the misconception, exit check required to
confirm repair worked, do not broaden scope.

`retrieval` — low-stakes recall from prior lessons, spaced from original
teaching, confidence building not new content, short and focused,
no new anchor required.

`transfer` — real-world application, novel contexts, learner applies method
without scaffolding, cold and extension questions only, no re-teaching.

**Support lenses:**

`high_load` — one idea per component, never combine explanation and example
in one block, visual before text where possible, DiagramBlock usage high,
more WorkedExampleCards before independent practice, warm questions heavily
scaffolded, short sentences, active voice, concrete before abstract.

`eal` — high visual density, every diagram element labelled explicitly,
DefinitionCard for every key term before first use, GlossaryInline
throughout practice, no idiom, no cultural references, mathematical
language over everyday language where both work.

`advanced` — compress scaffolding, one worked example maximum, formal
register, precise terminology, reach cold and transfer questions, include
at least one open-ended or proof-adjacent question, no hand-holding preamble.

## Piece 3 — Production Blueprint and WorkOrder Schemas

### What it is

The Production Blueprint is the instructional production plan the Lesson
Architect outputs. It is not the generated content — it is the contract
that tells each executor what to produce and what not to change.

WorkOrders are compiled from the blueprint — one per executor type. Each
executor reads only its WorkOrder. No executor sees the full blueprint.

### The relationship

```
ProductionBlueprint
    ↓  blueprint compiler (deterministic, no LLM)
WorkOrders
    ↓  executors produce
SectionContent / QuestionContent / VisualContent / AnswerKey
```

### Critical design boundary

The blueprint contains `content_intent` — a precise instruction for what
the executor should write. It does not contain generated prose.

```python
# Wrong — executor prose in blueprint
{
  "explanation_text": "Area means the amount of space inside a shape..."
}

# Right — intent for executor to fulfill
{
  "content_intent": "Explain that irregular shapes can be split into
   rectangles, using short sentences and the anchor L-shape as the example."
}
```

### Production Blueprint Pydantic schema

```
backend/src/blueprint/
  schema.py        # full ProductionBlueprint model + sub-models
  workorders.py    # all five WorkOrder models
  compiler.py      # compiles blueprint → WorkOrders (no LLM)
  validator.py     # validates blueprint completeness before teacher review
```

**Full schema:**

```python
from typing import Literal, Optional, Any
from pydantic import BaseModel, Field

Difficulty = Literal["warm", "medium", "cold", "extension", "transfer"]
VisualMode = Literal[
    "none", "diagram", "diagram_series", "image", "simulation"
]
ScaffoldingLevel = Literal["high", "medium", "low", "none"]
RegisterLevel = Literal["simple", "standard", "formal"]
ResourceType = Literal[
    "mini_booklet", "worksheet", "practice_set",
    "quiz", "exit_ticket", "revision_sheet", "lesson_pack"
]


class BlueprintMetadata(BaseModel):
    blueprint_id: str
    version: str = "3.0"
    created_for_generation_id: Optional[str] = None


class TeacherBriefSnapshot(BaseModel):
    year_group: str
    subject: str
    duration_minutes: Optional[int] = None
    free_text: str


class SignalSummary(BaseModel):
    topic: str
    subtopic: Optional[str] = None
    prior_knowledge: list[str] = Field(default_factory=list)
    learner_needs: list[str] = Field(default_factory=list)
    teacher_goal: str
    inferred_resource_type: ResourceType
    confidence: Literal["low", "medium", "high"]
    missing_signals: list[str] = Field(default_factory=list)


class AppliedLens(BaseModel):
    id: str
    label: str
    reason: str
    effects: list[str]  # concrete effects on THIS lesson, not generic rules


class LessonIntent(BaseModel):
    objective: str
    success_criteria: list[str]
    lesson_mode: Literal[
        "first_exposure", "consolidation", "repair",
        "retrieval", "transfer", "mixed"
    ]
    what_to_teach: list[str]
    what_not_to_teach: list[str] = Field(default_factory=list)
    likely_misconceptions: list[str] = Field(default_factory=list)
    fault_line: Optional[str] = None  # repair mode only


class LearnerProfilePlan(BaseModel):
    level_summary: str
    reading_load: Literal["low", "medium", "high"]
    language_support: Literal["none", "some", "high"]
    visual_support: Literal["low", "medium", "high"]
    pacing: Literal["short_chunks", "standard", "compressed"]


class RegisterPlan(BaseModel):
    level: RegisterLevel
    sentence_length: Literal["short", "mixed", "extended"]
    vocabulary_policy: Literal[
        "define_first", "define_in_context", "assume_known"
    ]
    tone: Literal["friendly", "clear_neutral", "exam_formal"]
    avoid: list[str] = Field(default_factory=list)


class AnchorExample(BaseModel):
    id: str
    label: str
    purpose: str
    reuse_scope: Literal[
        "entire_resource", "teaching_sequence_only",
        "practice_reference_only", "single_section"
    ]
    facts: dict[str, Any]
    correct_result: Optional[str] = None
    explanation: Optional[str] = None


class SourceOfTruth(BaseModel):
    anchor_examples: list[AnchorExample] = Field(default_factory=list)
    fixed_terms: list[str] = Field(default_factory=list)
    fixed_units: list[str] = Field(default_factory=list)
    formulas: list[str] = Field(default_factory=list)
    do_not_change: list[str] = Field(default_factory=list)


class ComponentPlan(BaseModel):
    component_id: str       # executor uses this
    teacher_label: str      # blueprint preview UI shows this
    content_intent: str
    uses_anchor_id: Optional[str] = None
    required: bool = True


class SectionVisualNeed(BaseModel):
    required: bool
    mode: VisualMode = "none"
    purpose: Optional[str] = None
    uses_anchor_id: Optional[str] = None
    placement: Optional[str] = None  # before | after | inline


class SectionPlanItem(BaseModel):
    id: str
    title: str
    order: int
    role: Literal[
        "orient", "explain", "define", "model",
        "practice", "check", "repair", "summarize", "revise"
    ]
    learning_intent: str
    components: list[ComponentPlan]
    visual: SectionVisualNeed
    register_notes: Optional[str] = None
    constraints: list[str] = Field(default_factory=list)


class QuestionPlanItem(BaseModel):
    id: str
    attaches_to_section_id: str
    difficulty: Difficulty
    purpose: str
    skill_target: str
    scaffolding: ScaffoldingLevel
    diagram_required: bool = False
    uses_anchor_id: Optional[str] = None
    expected_answer: str
    expected_working: Optional[str] = None
    student_facing_constraints: list[str] = Field(default_factory=list)


class VisualPlanItem(BaseModel):
    id: str
    attaches_to: str          # section_id or question_id
    mode: VisualMode
    purpose: str
    uses_anchor_id: Optional[str] = None
    must_show: list[str] = Field(default_factory=list)
    must_not_show: list[str] = Field(default_factory=list)
    labels_required: list[str] = Field(default_factory=list)
    consistency_locks: list[str] = Field(default_factory=list)
    print_requirements: list[str] = Field(default_factory=list)


class VisualStrategy(BaseModel):
    global_strategy: Literal[
        "low_visual", "selective_visuals", "high_visual_density",
        "same_anchor_sequence", "diagram_series_core"
    ]
    rationale: str
    visuals: list[VisualPlanItem] = Field(default_factory=list)


class ResourcePlan(BaseModel):
    resource_type: ResourceType
    title: str
    estimated_pages: Optional[int] = None
    print_first: bool = True
    student_facing: bool = True
    teacher_materials: list[str] = Field(default_factory=list)


class SupportAdaptation(BaseModel):
    id: str
    reason: str
    concrete_effects: list[str]


class ConsistencyRule(BaseModel):
    id: str
    rule: str
    check_method: Literal["deterministic", "llm_review", "manual_review"]


class AnswerKeyPlan(BaseModel):
    required: bool = True
    style: Literal["answers_only", "brief_explanations", "full_working"]
    include_question_ids: list[str] = Field(default_factory=list)
    notes: Optional[str] = None


class ReviewPlan(BaseModel):
    checks: list[ConsistencyRule]
    repair_policy: Literal[
        "surgical_only", "allow_section_regen"
    ] = "surgical_only"


class ProductionBlueprint(BaseModel):
    metadata: BlueprintMetadata
    teacher_brief: TeacherBriefSnapshot
    signal_summary: SignalSummary

    lesson_intent: LessonIntent
    learner_profile: LearnerProfilePlan
    lenses: list[AppliedLens]

    resource_plan: ResourcePlan
    register: RegisterPlan
    source_of_truth: SourceOfTruth

    section_plan: list[SectionPlanItem]
    question_plan: list[QuestionPlanItem] = Field(default_factory=list)
    visual_strategy: VisualStrategy

    support_adaptations: list[SupportAdaptation] = Field(default_factory=list)
    consistency_rules: list[ConsistencyRule] = Field(default_factory=list)
    answer_key_plan: AnswerKeyPlan
    review_plan: ReviewPlan
```

## Piece 4 — Lesson Architect Node

### What it is

A single Opus call with an extended thinking budget. It receives the teacher
signals, the Lectio manifest, and the full lens library. It reasons through
the complete lesson before committing to any output. It produces one thing:
a validated `ProductionBlueprint`.

This is the intelligence of the entire system. Everything else is execution
infrastructure built around it.

### Signal Extractor

Runs before the Lesson Architect. Fast, cheap model (Haiku). Reads the
teacher's free-text input and structured form fields, extracts structured
signals, and identifies what is missing.

```
backend/src/architect/
  signal_extractor.py
  clarification.py
  lesson_architect.py
  prompts/
    signal_extractor.txt
    clarification.txt
    lesson_architect.txt
```

**Signal Extractor output:**

```python
class ExtractedSignals(BaseModel):
    topic: str
    subtopic: Optional[str]
    year_group: str
    subject: str
    duration_minutes: Optional[int]
    prior_knowledge: list[str]
    learner_signals: list[str]       # confidence, reading level, EAL etc
    resource_intent: Optional[str]   # worksheet, booklet, quiz etc
    support_signals: list[str]       # ADHD, dyslexia, EAL mentions
    missing: list[str]               # what the architect still needs
    confidence: Literal["low", "medium", "high"]
```

### Clarification loop

Runs if `ExtractedSignals.confidence` is low or `missing` is non-empty.
Maximum two questions. Conversational, not a form. Stops as soon as the
architect has enough signal to plan with confidence.

```python
class ClarificationQuestion(BaseModel):
    question: str
    reason: str          # why this signal matters for planning
    optional: bool       # can the architect proceed without this?
```

The loop never asks about things the teacher already said. It only targets
genuine gaps.

### Lesson Architect system prompt — structure

The system prompt gives the Architect four things as its knowledge base:

**1. Its role:**
```
You are a master lesson designer. You receive a teacher's brief and produce
a complete instructional production plan — the Production Blueprint.

You hold the entire lesson in mind simultaneously. You make every
pedagogical decision before any content is written. You do not generate
content. You plan what content must be generated and why.
```

**2. The Lectio manifest** — injected in full as structured JSON. All 28
components with `phase`, `role`, `cognitiveJob`, `capabilities`,
`teacherLabel`. The Architect selects from this list. It cannot invent
components that do not exist in the manifest.

**3. The lens library** — injected in full. All lesson lenses and support
lenses. The Architect reads these as pedagogical principles to reason with,
not rules to follow mechanically.

**4. The blueprint schema** — injected as a JSON schema. The Architect knows
exactly what structure it must produce. Output is validated against this
schema before it leaves the node.

## Piece 5 — Surgical Repair Definition

### What it is

When the coherence reviewer flags a problem, only the broken WorkOrder is
repaired. Not the section. Not the lesson. Not the blueprint. Just the
specific output that failed the check.

### Why it needs defining now

Without an explicit repair policy, agents default to full regeneration.
Full regeneration is expensive, slow, and can introduce new inconsistencies
while fixing old ones. Surgical repair is the correct default.

### Repair targets

```python
class RepairTarget(BaseModel):
    type: Literal[
        "visual", "question", "answer_key",
        "section_component", "section_prose"
    ]
    id: str             # WorkOrder ID of the broken output
    reason: str         # what specifically is wrong
    blueprint_ref: str  # which blueprint field was violated
```

### Repair policy

```
bad visual          → regenerate VisualGeneratorWorkOrder only
bad question        → regenerate QuestionWriterWorkOrder slice only
bad answer          → recompute AnswerKeyWorkOrder entry only
bad section prose   → rewrite specific ComponentPlan only
bad register        → rewrite section with corrected register note

full regeneration   → only if teacher explicitly requests it
                      or if > 3 surgical repairs fail on same target
```

### What triggers a repair vs a full replan

```
Surgical repair:
  - Visual shows wrong dimensions
  - Question difficulty does not match blueprint
  - Answer key answer is wrong
  - Register too formal/informal for learner profile
  - Component missing from section

Full replan (rare — teacher decision only):
  - Teacher rejects the anchor example
  - Teacher changes the resource type
  - Teacher changes the lesson mode
```

### Verification checklist

```
□ CoherenceReviewWorkOrder produces RepairTarget list not just pass/fail
□ Each RepairTarget identifies exactly which WorkOrder to regenerate
□ Surgical repair never touches sections not referenced in RepairTarget
□ Full regeneration requires explicit teacher or system trigger
□ Repair loop has a maximum of 3 attempts before escalating to teacher
```
